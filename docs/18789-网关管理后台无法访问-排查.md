# 18789 端口 OpenClaw 网关管理后台无法访问 — 排查与解决

## 一、现象与原因

- **18789** 是 OpenClaw **Gateway** 的端口（WebSocket + HTTP 单端口复用），提供控制界面、`/healthz`、API 等。
- 当前网关进程**只监听本机回环地址**（`127.0.0.1` 和 `::1`），即默认 **bind=loopback**。
- 因此：
  - **本机**访问 `http://127.0.0.1:18789` 可以正常（如 `/healthz` 返回 200）。
  - **从其他机器**用 `http://<服务器IP>:18789` 访问会连不上，属于预期行为。

## 二、本机可访问性自检

在本机执行：

```bash
curl -s -o /dev/null -w "%{http_code}" http://127.0.0.1:18789/healthz
# 正常应输出 200
```

若本机也访问不了，请先确认：
- 网关进程是否在运行：`ss -tlnp | grep 18789` 或 `systemctl status openclaw-gateway`
- 是否有本地防火墙/代理拦截 127.0.0.1:18789

## 三、从其他机器访问 18789（需改绑定）

若需要从**局域网或外网**访问 18789 的网关管理后台/Control UI，必须让 Gateway 监听所有网卡（或指定网卡），并**务必配置鉴权**。

### 1. 配置鉴权（必须）

在 `~/.openclaw/openclaw.json`（或 `OPENCLAW_STATE_DIR` 下的配置）中确保有 Token：

```json5
{
  "gateway": {
    "auth": { "mode": "token", "token": "你的强密码或随机 Token" }
  }
}
```

或使用环境变量 `OPENCLAW_GATEWAY_TOKEN`。  
**未配置鉴权时，bind 改为 lan 会触发安全策略，网关可能拒绝启动。**

### 2. 将绑定改为 LAN

在 `openclaw.json` 的 `gateway` 中设置：

```json5
{
  "gateway": {
    "bind": "lan",
    "port": 18789
  }
}
```

或用 CLI：

```bash
openclaw config set gateway.bind lan
```

### 3. 重启网关使配置生效

- **systemd**：`sudo systemctl restart openclaw-gateway`
- **手动运行**：先停止当前进程，再执行 `openclaw gateway --port 18789`（或你原来的启动方式）

重启后，Gateway 会监听 `0.0.0.0:18789`，即可通过 `http://<本机IP>:18789` 从其他机器访问。访问时需在 Control UI 或 API 请求中携带上述 Token。

## 四、不改绑定时的替代方式（推荐用于远程）

若希望保持 **bind=loopback**（更安全），可从其他机器通过 **SSH 隧道**访问 18789：

```bash
# 在要访问的客户端机器上执行：
ssh -N -L 18789:127.0.0.1:18789 user@<网关所在服务器IP>
```

然后在客户端浏览器打开 `http://127.0.0.1:18789`，即等同于访问服务器上的网关管理后台。

## 五、与管理后台 8080 的区别

| 端口  | 用途                         | 默认监听     |
|-------|------------------------------|--------------|
| **8080** | AmyClaw **管理设置页**（Token、飞书、系统密码等） | `0.0.0.0:8080` |
| **18789** | OpenClaw **Gateway**（Control UI、API、WebSocket） | `127.0.0.1:18789` |

若只是要改 Token、飞书、系统密码等，访问 **http://&lt;本机IP&gt;:8080** 即可，无需开放 18789。

### 日志里「Control UI build failed」与飞书配置是否有关？

- **一般无关。** 飞书、大模型、Token 等由 **8080 管理页** 写入 **`/root/.openclaw/openclaw.json`**，依赖的是 **`openclaw-management`** 服务，**不依赖** 18789 上的 Control UI 静态资源是否构建成功。
- **18789** 上的 **Control UI** 需要包内 **`dist/control-ui/index.html`**（由开发机构建时执行 **`pnpm ui:build`** 生成）。量产 **`/opt/amyclaw`** 若未包含该目录，网关启动时会尝试自动执行 `ui:build`，在目标机上常因缺少完整前端依赖而失败，日志可能出现 **`Control UI build failed`**；此时 **Gateway 仍可监听 18789**，API/WebSocket/飞书通道照常（只要配置正确）。
- **若飞书保存后仍不生效**：请检查 **`openclaw.json`** 里是否已有 `channels.feishu.accounts` 与 `enabled: true`，以及 **`plugins.entries.feishu.enabled`** 是否为 `true`；与 Control UI 构建无直接关系。
- **修复 18789 网页**：在**母机**仓库 `openclaw` 下执行 **`pnpm ui:build`** 后重新 **`deploy-to-opt.sh`**（脚本已可在 deploy 前自动执行 `ui:build`），再重启网关。

---

**总结**：18789 无法从外网/局域网访问是因为默认只监听本机。需要远程访问时，要么在配置鉴权后设置 `gateway.bind=lan` 并重启网关，要么使用 SSH 隧道访问，无需改 bind。
