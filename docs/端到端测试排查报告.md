## 二、401 原因分析

- 脚本和本机配置中使用的鉴权方式：
  - **openclaw.json**：`gateway.auth.token` 已配置（48 位）。
  - **.env**：`OPENCLAW_GATEWAY_TOKEN` 已设置（48 位）。
- 使用上述两种 Token 分别请求 `POST /v1/chat/completions`，均返回 **401**。
- **结论**：当前在 18789 端口提供服务的 Gateway 进程**不是**用本机 `OPENCLAW_STATE_DIR` 下的 `openclaw.json` / `.env` 启动的，因此和本机的 Token 不一致。  
  要端到端跑通，需要二者一致（见下文「修复建议」）。

## 三、飞书端到端：如何看日志

飞书「发消息」的完整链路在 **Gateway 进程内**完成。要排查「发消息不回」，必须在**运行 Gateway 的那台机器、该进程的终端或日志**里看输出。

### 1. 在运行 Gateway 的终端/日志中搜关键词（按顺序）

| 顺序 | 关键词 | 含义 |
|------|--------|------|
| 1 | `feishu: starting` 或 `starting feishu[default]` | 飞书通道是否启动 |
| 2 | `feishu[...]: WebSocket client started` 或 `starting Webhook` | 长连接/Webhook 是否建好 |
| 3 | `feishu[...]: received message from` | 是否收到飞书消息事件 |
| 4 | `did not mention bot` | 群消息是否因未 @ 被跳过 |
| 5 | `dispatching to agent` | 是否进入大模型 |
| 6 | `dispatch complete (queuedFinal= true, replies=` | 是否有回复要下发 |
| 7 | `Feishu send failed` 或 `Feishu reply failed` | 调用飞书发消息 API 是否报错 |

### 2. 建议操作步骤

1. 在本机用**同一** `OPENCLAW_STATE_DIR`（和同一 `openclaw.json`、`.env`）启动 Gateway，例如：
   ```bash
   export OPENCLAW_STATE_DIR=/root/.openclaw   # 或你的实际目录
   openclaw gateway
   ```
2. 在飞书里给机器人发一条消息（群聊需先 @ 机器人，或已设 `requireMention: false`）。
3. 立刻在 Gateway 终端中查找上表中的 1～7 步关键词，确定断在哪一环。
4. 详细说明见：`docs/热加载与无回复排查.md` 第六节「飞书发消息端到端全链路检查」。

## 四、大模型返回空内容（未生成回复根因）

当 Chat Completions 返回 HTTP 200 但 `content` 为 `"No response from OpenClaw."`，且日志出现：

```text
[agent/embedded] embedded run produced no payloads: ... assistantTextsLen=0 lastAssistantStopReason=stop lastAssistantError=none lastAssistantRawTextLen=0
```

表示 **模型 API 正常结束但返回内容为空**（`lastAssistantRawTextLen=0`）。主/次模型不冲突：fallback 仅在主模型**抛错**时触发，主模型返回空内容时不会自动换 fallback。

**建议**：
1. **单独测大模型**：`OPENCLAW_STATE_DIR=/root/.openclaw ./scripts/test-llm-direct.sh`，确认是否同样得到「No response from OpenClaw」。
2. **临时改用 fallback 模型**：在管理端或 `openclaw.json` 将 `agents.defaults.model.primary` 改为当前 fallback（如 `moonshot/kimi-k2.5`），保存并触发热加载或重启 Gateway，再测 Chat Completions 与飞书回复；若 fallback 有正常内容，则问题在主模型/API（idatabase 的 Key、配额、或模型 id `gemini-3.1-flash-lite-preview` 可用性）。
3. **查 idatabase**：确认 `IDATABASE_API_KEY` 有效、未超配额、网络可达；必要时联系 API 提供方确认该模型是否返回空 body。

详见：`docs/未生成回复与OPT-DEV交叉排查.md` 第四节、第五节。

- **方案 A**：用本机配置启动当前要测的 Gateway  
  - 在要排查的机器上执行：`OPENCLAW_STATE_DIR=/root/.openclaw openclaw gateway`（或你的实际目录）。

完成后重新执行：

```bash

./scripts/e2e-test-feishu-and-model.sh
```
