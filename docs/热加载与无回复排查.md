# 配置热加载与「大模型/飞书无回复」排查

## 一、热加载是否已生效？

### 1. 热加载链路（已实现）

- **管理端保存**：点击「保存全部配置」→ 写入 `openclaw.json` 与 `.env` → 请求 Gateway **POST /reload-config**（Bearer 鉴权）。
- **Gateway**：收到 `/reload-config` 后执行一次配置重载（重读 `openclaw.json`，计算 diff，执行 hot 或提示 restart）。
- **热加载时**：会重新执行 **loadDotEnv()**，把 `OPENCLAW_STATE_DIR/.env` 再读进 `process.env`，因此**管理端刚写入的 API Key 会在本次热加载后生效**，无需重启 Gateway。

### 2. 如何确认热加载已触发？

- **看 Gateway 日志**：热加载会打 `config change detected; evaluating reload (...)`，以及 `config change applied` 或 `config reload disabled`。
- **看管理端**：保存成功提示为「配置已更新；若 Gateway 已运行将自动热加载」。若管理端与 Gateway 的 **OPENCLAW_STATE_DIR** 或 **OPENCLAW_GATEWAY_URL** 不一致，或 Token 不一致，则 `POST /reload-config` 会失败（管理端仅打 warn 日志，不阻塞保存）。

### 3. 同源要求

- 管理端与 Gateway 必须使用**同一** `OPENCLAW_STATE_DIR`（默认 `~/.openclaw`）。
- 配置只来自 `openclaw.json`，密钥只来自 `.env`；管理端写入的就是这两个文件，热加载会重读它们。

## 二、为什么大模型没有回复？

### 1. 常见原因

| 原因 | 说明 | 处理 |
|------|------|------|
| **API Key 未生效** | 在管理端新填的 Key 写在 `.env`，热加载会重新 loadDotEnv，一般会生效；若未生效，检查是否同源、是否保存成功、Gateway 是否收到 reload。 | 保存后看 Gateway 日志是否有 config change applied；仍不行则**重启 Gateway** 一次。 |
| **无效的令牌 / 401** | 若上游返回「无效的令牌」，说明 Key 错误或已过期。 | 在管理端或 `.env` 中核对并更新对应 Key（如 MOONSHOT_API_KEY、IDATABASE_API_KEY）。 |
| **模型/路由配置** | agents.defaults.model 或 fallbacks 未配、或指向不可用模型。 | 检查 `openclaw.json` 里 `agents.defaults.model`、`models`、对应 provider 的 apiKey 引用。 |
| **热加载未包含 .env** | 旧版本热加载未重读 .env。 | 已修复：热加载时会调用 `loadDotEnv({ quiet: true })`，新 Key 会生效。 |

### 2. 建议自检

- 管理端「大模型」页确认 Key 已保存，且「解析后的密钥」与预期一致。
- 在管理端 GET `/api/status` 看 `stateDir`、`configPath`、`envPath`，确认与 Gateway 一致。
- 若刚保存 Key：先点保存 → 等 1～2 秒 → 再发一条消息测大模型。

## 三、为什么飞书没有回复？

### 1. 群聊必须 @ 机器人（默认）

- 群聊下默认 **requireMention: true**：只有**@ 了机器人**才会回复；未 @ 则直接 return，并打日志 `message in group ... did not mention bot`。
- 若希望群内任意消息都回复，可在对应群配置或全局飞书配置中设 **requireMention: false**（需热加载或重启后生效）。

### 2. @ 了仍不回复的可能原因

| 原因 | 说明 | 处理 |
|------|------|------|
| **未识别到 @** | 飞书事件里 `mentions` 可能为空或只有 user_id，与当前用 open_id 的判断不一致。 | 看 Gateway 日志里 `feishu[...]: message in group ... did not mention bot (message_type=... mentions=... botOpenId=...)`，确认 botOpenId 与 mentions 是否匹配。 |
| **飞书通道未热加载** | 修改的是 `channels.feishu`，热加载会触发 **restart-channel:feishu**，会重启飞书通道。 | 保存后看日志是否有 channel 重启；若改了 app_id/app_secret，一般会重启飞书通道并重连。 |
| **app_id / app_secret 错误** | 应用未启用或凭证错误，收不到事件或发不出消息。 | 在飞书开放平台核对应用启用、权限、app_id/app_secret；管理端可测「飞书应用凭证」。 |
| **未启用或未保存** | 飞书账号 disabled 或未保存。 | 管理端飞书配置里确认已启用并保存；保存后会触发热加载。 |

### 3. 建议自检

- 群聊中**先 @ 机器人**再发一句话，看是否有回复。
- 不需要 @ 才回复：在配置里设 `requireMention: false` 并保存，等热加载生效。
- 看 Gateway 日志中 feishu 相关行（如 `did not mention bot`、restart channel、错误码）。

---

## 四、热加载会做哪些事？

- **仅改 openclaw.json / .env**：  
  - 热加载会重读 `openclaw.json` 和 `.env`（通过 loadDotEnv），然后根据 diff 执行：
  - **reload-hooks**（hooks 配置）
  - **restart-heartbeat**（模型/agents 相关）
  - **restart-cron**、**restart-browser-control**、**restart-health-monitor** 等
  - **restart-channel:feishu**（修改了 `channels.feishu` 时）
- **改了 plugins / gateway / discovery 等**：会要求**重启 Gateway**（热加载会打 restart 原因并可能发 SIGUSR1）。

---

## 五、总结

1. **热加载已生效**：保存后管理端会调 Gateway `POST /reload-config`，且热加载时会重读 `.env`，大模型新 Key 会生效。
2. **大模型无回复**：优先查 Key 是否生效（保存→热加载→再试）、令牌是否有效、模型/路由配置。
3. **飞书无回复**：优先确认是否 @ 了机器人、是否设了 `requireMention: false`、飞书通道是否在热加载后重启、app_id/app_secret 是否正确。

## 六、飞书发消息端到端全链路检查

当「飞书发消息还是不回」时，按下面顺序逐环排查，每环都通过再看下一环。

### 链路概览

```
飞书服务器 → [1] 长连接/Webhook → [2] 事件分发 → [3] handleFeishuMessage
    → [4] 鉴权/@/群策略 → [5] dispatchReplyFromConfig(大模型) → [6] dispatcher.deliver
    → [7] sendMessageFeishu / 卡片 → [8] 飞书发消息 API → 用户看到回复
```

### 1. 飞书通道是否在跑？（收事件）

- **代码位置**：`extensions/feishu/src/monitor.ts` → `monitorFeishuProvider`，由 Gateway `server-channels` 的 `startAccount` 调用。
- **检查**：
  - Gateway 启动后日志应有：`feishu: starting N account(s)`、`starting feishu[default] (mode: websocket)` 或 webhook 提示。
  - 热加载 `channels.feishu` 后会执行 `restart-channel:feishu`，日志应有 channel 重启（见「四、热加载会做哪些事」）。
- **未跑常见原因**：
  - **`channels.feishu.enabled` 为 false**、未配置 `accounts.default.appId/appSecret`。
  - **`plugins.entries.feishu.enabled` 为 false**（与上面不同）：此时飞书**扩展不会加载**，即使用户在界面里打开了飞书通道，也**不会出现** `starting feishu[...]`，**收不到任何消息**。请在 `openclaw.json` 中设为 `true` 并**重启 Gateway**（或保存后若触发了需整进程重启的变更，以日志为准）。  
  - **代码侧加固**：OpenClaw 已在配置归一化（`applyPluginAutoEnable`）中修复：当 **`channels.<通道>.enabled=true` 但 `plugins.entries.<通道>.enabled=false`** 时，会自动对齐为启用插件条目（与 `enablePluginInConfig` 行为一致）。**合并该修复后需重新构建/部署 openclaw**；已部署的旧机仍可手动改 JSON 或依赖保存配置时的归一化逻辑。
  - Gateway 未正确加载插件。

### 2. 事件是否到达并进入处理？

- **代码位置**：`extensions/feishu/src/monitor.account.ts` → `registerEventHandlers` → `dispatchFeishuMessage` → `handleFeishuMessage`。
- **检查**：在 Gateway 日志中搜 `feishu[default]: received message from` 或 `feishu[default]: processing merge_forward`。若完全没有任何 `feishu[...]:` 消息日志，多半是：
  - 长连接未建成功（app_id/app_secret 错误、网络或飞书开放平台配置），或
  - Webhook 未收到请求（URL、验证未通过）。
- **长连接模式说明（不需要配域名）**：使用「长连接」时，是 **Gateway 主动连飞书**，不需要公网域名或回调 URL。若日志里已有 `[ws] ws client ready`、`feishu[default]: WebSocket client started`，说明长连接已建立；若仍然**没有** `received message from`，说明飞书侧**没有通过长连接推送消息事件**。请到 **飞书开发者后台** 确认：
  1. **事件与回调** → **订阅方式** 选择 **「使用 长连接 接收事件/回调」**（不是「请求地址」）。
  2. **订阅事件** 中包含 **「接收消息」**（如 `im.message.receive_v1`）。
  3. 应用类型为 **自建应用**（长连接仅对自建/飞书应用开放）。
  4. 机器人已加入目标群聊或与用户存在单聊；在**该会话内**发一条消息才会触发事件。

### 3. 是否被「未 @ 机器人」拦截？（仅群聊）

- **代码位置**：`extensions/feishu/src/bot.ts` 中 `checkBotMentioned`，未 @ 时直接 return 并打日志。
- **检查**：日志里是否有：`feishu[...]: message in group ... did not mention bot (message_type=... mentions=... botOpenId=...)`。
- **处理**：群内先 @ 机器人再发；或配置 `requireMention: false` 并热加载/重启飞书通道。

### 4. 是否被 DM/群策略拦截？

- **代码位置**：同文件中的 `dmPolicy`、`groupPolicy`、`groupAllowFrom`、`allowFrom` 等。
- **检查**：日志中是否有 `blocked unauthorized sender`、`group not in groupAllowFrom`、`group is disabled` 等。私聊若为 pairing 模式会发配对说明，不会直接回复内容。

### 5. 大模型是否被调用并返回内容？

- **代码位置**：`handleFeishuMessage` → `createFeishuReplyDispatcher` → `withReplyDispatcher` → `dispatchReplyFromConfig`（`src/auto-reply/reply/dispatch-from-config.ts`）→ `getReplyFromConfig`（大模型）。
- **检查**：日志中是否有 `feishu[...]: dispatching to agent`、`dispatch complete (queuedFinal=..., replies=...)`。若 `replies=0` 且无后续发送日志，可能是：
  - 大模型 API 报错（见「二、为什么大模型没有回复」），
  - 或 dispatch 内部逻辑跳过了回复（如重复、跳过等）。
- **建议**：先在同一环境用「一、热加载」确认大模型能正常返回，再发飞书消息。

### 6. 回复是否进入 deliver 并调用发送？

- **代码位置**：`extensions/feishu/src/reply-dispatcher.ts` 的 `deliver` 回调 → `sendMessageFeishu` / `sendMarkdownCardFeishu` / `sendMediaFeishu`（`send.ts`）。
- **检查**：若 5 有 `replies>=1` 但用户端仍无消息，看是否有 `Feishu send failed` / `Feishu reply failed` 或飞书 API 错误码（如 99991672 权限、230011 等）。这些会从 `send-message.ts` 的 `assertFeishuMessageApiSuccess` 抛出并打到 error 日志。

### 7. 飞书发消息 API 是否成功？

- **代码位置**：`extensions/feishu/src/send-message.ts`（reply 或 create），成功时 `code === 0`。
- **检查**：任何非 0 的 code 都会以 `Feishu send failed: msg` 或 `Feishu reply failed: msg` 形式抛出。常见：权限不足、receive_id 类型错误、消息被风控等。需在飞书开放平台核对应用权限与可用范围。

### 8. 快速日志关键词汇总

在 Gateway 日志中可依次搜：

| 关键词 | 说明 |
|--------|------|
| `feishu: starting` | 飞书通道启动 |
| `feishu[...]: WebSocket client started` / `starting Webhook` | 长连接或 Webhook 已起 |
| `feishu[...]: received message from` | 收到消息并进入处理 |
| `did not mention bot` | 群消息未 @ 被跳过 |
| `blocked unauthorized` / `not in groupAllowFrom` | 被策略拦截 |
| `dispatching to agent` | 进入大模型调度 |
| `dispatch complete (queuedFinal= true, replies=1)` | 有最终回复并会下发 |
| `Feishu send failed` / `Feishu reply failed` | 调用飞书发消息 API 失败 |

按 1→8 顺序排查，可快速定位断在「收事件、鉴权、大模型、发消息」中的哪一环。

## 七、量产机一键自检（SSH 到设备后执行）

在 **运行 Gateway 的机器**上（如 `OPENCLAW_STATE_DIR=/root/.openclaw`），**量产镜像内**已随 OEM 放入：

```bash
sudo bash /opt/amyclaw/oem/diagnose-openclaw-gateway-feishu.sh
```

母机开发仓库中为 **`/mnt/disk/amyclaw/scripts/diagnose-openclaw-gateway-feishu.sh`**（由 `deploy-to-opt.sh` 复制到 `/opt/amyclaw/oem/` 后再打 ISO）。若镜像较旧无此文件，可从母机 `scp` 该脚本到设备再执行。

脚本会检查：`openclaw-gateway` / `openclaw-management` 是否 active、`openclaw.json` 与 `.env` 是否存在及修改时间、飞书开关与账户键名、`.env` 中 Key 是否非空（**不打印密钥原文**）、端口监听、以及 journal 中与热加载/飞书相关的行。

若 **无法 SSH**：在设备本机控制台登录后同样可运行上述脚本；并确认 `sshd` 是否允许所用账号（部分环境禁止 `root` 密码登录，需用 `amyclaw` + `sudo`）。若提示 **amyclaw is not in the sudoers file**，说明镜像未把 `amyclaw` 加入 `sudo` 组：母机须用最新 **`set-oem-system-passwords.sh`** 重打 ISO；现场恢复见 **`eggs-量产克隆与使用说明.md`**「现场被锁」。
