# 全新机器恢复完整指南（AmyClaw Live 系统）

本文档说明如何在一台**全新的电脑**上，从 AmyClaw 备份 ISO 恢复并运行完整系统（Ubuntu + OpenClaw Gateway + 管理前后端）。适用于 **v1.01** 及之后版本。

---

## 一、准备阶段

### 1.1 你需要的东西

| 项目 | 说明 |
|------|------|
| **ISO 文件** | 例如 `amyclaw_v1.01_20260316_0601.iso`（约 3GB），从源机 `/mnt/backup` 或 download 共享获取 |
| **目标机器** | 任意 x86_64 电脑，建议 ≥4GB 内存、≥20GB 可用磁盘 |
| **启动介质** | 至少 4GB 的 U 盘（用于制作 Live 启动盘），或已配置的 PXE 网络启动 |

### 1.2 镜像内已包含内容

- **系统**：基于 Ubuntu 22.04 的 Live 系统，可安装到硬盘。
- **应用**：OpenClaw Gateway（端口 18789）、管理 Web 页（端口 8080），位于 **/opt/amyclaw**（仅编译后文件，无源码）。
- **工作区指向**：**克隆/恢复后与开发机不同**。开发机通常从仓库 **/mnt/disk/amyclaw**（DEV 路径）运行，systemd 单元里 `WorkingDirectory` 指向该路径；打包时 **`docs/penguins-eggs/deploy-to-opt.sh`** 会把应用部署到 **/opt/amyclaw**（opt 路径）并写入指向 `/opt/amyclaw` 的 systemd 单元。因此恢复出来的机器上，服务从 **/opt/amyclaw** 读代码与静态文件（如管理页样式），不会读 `/mnt/disk/amyclaw`。**测试可在 DEV 或 opt 任一路径做；测试通过后必须执行 `deploy-to-opt.sh` 将 DEV 同步到 opt，再打包，否则量产仍是旧版。** 详见 [dev与opt路径与同步.md](./dev与opt路径与同步.md)。若在开发机改了 `openclaw-management/public/index.html` 或 `server.js` 等，需重新执行部署脚本并重新打包，或在恢复机上单独同步到 `/opt/amyclaw/openclaw-management/`。**模型与记忆配置自测**：在仓库根执行 `OPENCLAW_STATE_DIR=/root/.openclaw node scripts/self-test-model-memory.js` 可校验主模型、记忆接入与 DEV/opt 路径是否一致。
- **预置账号**（量产镜像）：  
  - 用户 **amyclaw**，密码 **amyclaw20260315**  
  - **root** 的密码为 **amyclaw20260315**（用户名仍是 `root`，**不是** root/root）  
  安装到硬盘后由 Krill 写入新系统；母机打 ISO 前须已执行 `set-oem-system-passwords.sh` 并部署 `krill-amyclaw.yaml`（见 `mother-image-prepare-for-amyclaw-iso.sh`）。

### 1.3 镜像内不包含（需恢复后配置）

- `/root/.openclaw`（Token、API Key、openclaw.json 等）——出于安全已排除，需在恢复后通过管理页或模板初始化。
- 个人数据、其他机器的敏感文件。

## 二、把 ISO 弄到目标机或能启动的位置

任选一种方式，让目标机能够**从该 ISO 启动**。

### 方式 A：U 盘启动

1. 在一台已装系统的电脑上，用 **Ventoy** 或 **Rufus**（Windows）/ **dd**（Linux）把 ISO 写入 U 盘。
   - **Ventoy**：将 ISO 文件直接拷贝到 U 盘即可，无需“刻录”。
   - **dd**（Linux，会清空 U 盘）：
     ```bash
     # 确认 U 盘设备名，例如 /dev/sdb
     sudo dd if=amyclaw_v1.01_20260316_0601.iso of=/dev/sdb bs=4M status=progress oflag=sync
     ```
2. 将 U 盘插入**全新机器**，开机进 BIOS/UEFI，设置从该 U 盘启动。

### 方式 B：网络启动（PXE）

若你已有 PXE 服务器，将 ISO 内 `live/vmlinuz*`、`live/initrd.img*`、`live/filesystem.squashfs` 放到 PXE 可访问路径，并配置引导菜单指向该内核与 initrd，目标机从网卡启动即可。

### 方式 C：虚拟机

在 VMware / VirtualBox / KVM 中新建虚拟机，光驱挂载该 ISO，从光驱启动。

## 三、从 ISO 启动并安装到硬盘

### 3.1 启动到 Live 系统

1. 目标机从 U 盘（或 PXE/虚拟机）启动，选择从 Live 系统引导。
2. 进入 Live 桌面后，可先试用；要**永久安装**则继续下一步。

### 3.2 将系统安装到硬盘（二选一）

#### 方式一：命令行安装（推荐，无需图形）

在 Live 系统里打开终端，执行：

```bash
sudo eggs krill
```

按提示选择：

- 安装目标磁盘与分区（可整盘或手动分区）；
- 主机名、用户名等（若提示）；  
安装过程会复制当前 Live 系统到硬盘（含 /opt/amyclaw 和预置账号），并安装引导器。

#### 方式二：图形安装（若镜像带 Calamares）

- 在 Live 桌面点击「安装」图标，或终端执行：
  ```bash
  sudo eggs calamares --install
  ```
- 按 Calamares 向导选择语言、时区、磁盘分区、用户信息，完成安装。

### 3.3 重启到已安装系统

1. 安装完成后，按提示**取出 U 盘**（若用 U 盘），选择重启。
2. 在 BIOS 中将**硬盘**设为第一启动项。
3. 启动后进入的是已安装到硬盘的 Ubuntu，而不是 Live；root/amyclaw 密码与 1.2 节一致。

## 四、首次进入系统后的必做配置

以下步骤在**已安装好的系统**上执行（用 root 或带 sudo 的用户）。

### 4.1 确认预置账号（v1.01）

v1.01 镜像已包含用户 **amyclaw** 和 **root**，密码均为 **amyclaw20260315**。  
若需修改密码：

```bash
sudo passwd root
sudo passwd amyclaw
```

### 4.2 确认 Node.js（镜像通常已带）

OpenClaw 与管理端需要 Node.js ≥18。若未安装或版本过低：

```bash
sudo apt update
sudo apt install -y nodejs npm
node -v   # 建议 ≥18
```

### 4.3 确认应用路径与依赖（v1.01）

应用已在 **/opt/amyclaw**，一般无需再拷贝。若管理端缺少依赖，可补装：

```bash
sudo npm install -C /opt/amyclaw/openclaw-management 2>/dev/null || true
```

### 4.4 初始化 OpenClaw 状态目录（必做）

镜像中**不包含** `/root/.openclaw`，需在目标机创建并放入预设配置（无密钥）。当前模板为 **google 专用**（idatabase v1beta + google-generative-ai，主模型 `google/gemini-3.1-pro-preview`；记忆为 openai 兼容 + idatabase v1 + `text-embedding-005`），API Key 通过环境变量引用，需在 `.env` 中配置 **IDATABASE_API_KEY**（主模型与向量共用同一 Key 即可）：

```bash
export OPENCLAW_STATE_DIR=/root/.openclaw
sudo mkdir -p "$OPENCLAW_STATE_DIR"
sudo cp -r /opt/amyclaw/openclaw/production-template/state/* "$OPENCLAW_STATE_DIR/"
# 若有 .env.example，复制为 .env，并填写 IDATABASE_API_KEY（主模型与记忆向量共用）
sudo cp "$OPENCLAW_STATE_DIR/.env.example" "$OPENCLAW_STATE_DIR/.env" 2>/dev/null || true
# 编辑 .env，增加一行：IDATABASE_API_KEY=你的key
```

### 4.5 启用并启动服务

v1.01 的 systemd 单元已指向 `/opt/amyclaw`，安装到硬盘后通常已在 `/etc/systemd/system/`，直接启用并启动即可：

```bash
sudo systemctl daemon-reload
sudo systemctl enable openclaw-gateway openclaw-management
sudo systemctl start openclaw-gateway openclaw-management
```

检查状态：

```bash
sudo systemctl status openclaw-gateway openclaw-management
```

### 4.6 验证

- **Gateway**：本机执行 `curl -s http://127.0.0.1:18789` 有响应即正常。
- **管理页**：浏览器访问 `http://<本机IP>:8080`：
  - 首次使用可在页面中「自动生成」或手动填写 **OpenClaw Gateway Token**，保存后即与 Gateway 一致。
  - 在管理页中配置飞书、模型 API（如 idatabase）、向量、Minimax 等；API Key 会写入 `OPENCLAW_STATE_DIR/.env`，不写进 openclaw.json。
  - **飞书 E2E 自检**：重启网关后执行 `sudo journalctl -u openclaw-gateway -n 50 --no-pager | grep feishu`，应看到 `bot open_id resolved`、`ws client ready`；在飞书群/单聊中 @机器人 发一条消息，日志中应出现 `dispatching to agent` 与 `dispatch complete`。

## 五、安装后可选配置

- **防火墙**：若需从其他机器访问 8080/18789，在 `ufw` 或 iptables 中放行相应端口。
- **开机自启**：上述 `systemctl enable` 已设置，重启后 Gateway 与管理页会自动启动。
- **HTTPS/反向代理**：若要对管理页做 HTTPS 或域名访问，可在本机配置 Nginx/Caddy 等反向代理。

## 六、故障排查

| 现象 | 处理建议 |
|------|----------|
| **管理页显示「Gateway 未响应」** | 表示 Gateway 未启动或无法连接。执行 `sudo systemctl status openclaw-gateway` 查看状态；若为 **activating (auto-restart)** 或 **exit-code**，看崩溃原因：`sudo journalctl -u openclaw-gateway -n 50 --no-pager`。若为 **gateway.mode is unset** 或配置不完整，在仓库根执行 `OPENCLAW_STATE_DIR=/root/.openclaw bash scripts/fix-gateway-config-amyclaw.sh`（旧名 `fix-gateway-config-on-x1.sh` 仍可用）（会补全 `gateway.mode=local` 与 `gateway.remote.token`），然后 `sudo systemctl restart openclaw-gateway`。若为 Config invalid（如未知 channel id），执行 `openclaw doctor --fix` 或按报错删除无效键后重启。 |
| Gateway 或管理页无法访问 | `sudo systemctl status openclaw-gateway`、`openclaw-management` 看报错；确认 `WorkingDirectory=/opt/amyclaw/...`、`OPENCLAW_STATE_DIR=/root/.openclaw` 正确；必要时在 `/opt/amyclaw/openclaw-management` 下执行 `sudo npm install`。 |
| 管理页样式/前端刷新无变化 | 恢复机服务从 **/opt/amyclaw** 读文件，与开发机 **/mnt/disk/amyclaw** 是两套；在开发机改的 `public/index.html` 等需重新执行 `deploy-to-opt.sh` 并重新打包，或在恢复机上把改动文件拷到 `/opt/amyclaw/openclaw-management/public/` 后重启 `openclaw-management`。 |
| 管理页提示 Token 错误 | 在管理页重新生成或填写 Token 并保存，确保与 Gateway 共用同一状态目录（`/root/.openclaw`）；或检查 `openclaw.json` 中 `gateway.auth.token`。 |
| **openclaw CLI 报 Gateway not reachable / token_mismatch** | CLI 连接网关需 `gateway.remote.token` 与网关鉴权一致。在状态目录同机执行：`bash scripts/sync-gateway-remote-token.sh`（会从 `.env` 或 `openclaw.json` 取 token 并写入 `gateway.remote.token`）；或手动把管理页/`.env` 中的 Gateway Token 填入 `openclaw.json` 的 `gateway.remote.token`，与 `gateway.auth.token`（或 OPENCLAW_GATEWAY_TOKEN）一致。 |
| 依赖报错 / 模块找不到 | 在 `/opt/amyclaw/openclaw` 与 `/opt/amyclaw/openclaw-management` 下分别执行 `sudo npm install`（或 pnpm install，若镜像内带 pnpm）。 |
| **日志反复出现 duplicate plugin feishu / plugins.allow is empty** | 状态目录下存在 `extensions/feishu` 与内置 feishu 重复。可删除重复副本：`sudo rm -rf /root/.openclaw/extensions/feishu` 并重启；但若删除后网关报 **unknown channel id: feishu**，说明本机依赖该副本，需从安装目录恢复：`sudo mkdir -p /root/.openclaw/extensions && sudo cp -r /opt/amyclaw/openclaw/extensions/feishu /root/.openclaw/extensions/`，并在 `openclaw.json` 的 `plugins` 中加 `"allow": ["feishu"]`，再重启网关。 |
| **网关报 Config invalid / unknown channel id: feishu** | 配置中有 `channels.feishu` 但网关找不到 feishu 插件。从安装目录恢复：`sudo mkdir -p /root/.openclaw/extensions && sudo cp -r /opt/amyclaw/openclaw/extensions/feishu /root/.openclaw/extensions/`；在 `openclaw.json` 的 `plugins` 中加 `"allow": ["feishu"]`（若没有 `plugins` 则加 `"plugins": { "allow": ["feishu"] }`）；然后 `sudo systemctl restart openclaw-gateway`。 |
| **右下角「重启网关」按钮报 Not found** | 恢复机管理服务从 **/opt/amyclaw** 读代码，若该目录下 `server.js` 未含 `POST /api/gateway-restart` 会返回 404。处理：先在本机执行 `sudo systemctl restart openclaw-management` 并**强制刷新管理页**（Ctrl+Shift+R）；若仍 404，从开发机把最新 `openclaw-management/server.js` 与 `openclaw-management/public/index.html` 拷到 `/opt/amyclaw/openclaw-management/` 对应位置后重启 management。开发机可执行 `bash scripts/fix-restart-and-feishu.sh`（会编译 openclaw、若存在 /opt/amyclaw 则同步管理端、并重启网关与管理服务）。 |
| **飞书发消息无回复** | 若主模型为 google/idatabase（Gemini），需网关使用含「Google 工具 schema 清理」的构建，否则易出现 400（patternProperties/const）或 no-payloads。在**开发机**执行：`bash scripts/fix-restart-and-feishu.sh`（会编译并同步到 /opt/amyclaw 若存在、再重启网关）；或先 `cd openclaw && npm run build`，再 `sudo bash docs/penguins-eggs/deploy-to-opt.sh` 与 `sudo systemctl restart openclaw-gateway`。校验修复是否生效：`bash scripts/verify-google-tools-fix.sh`。 |
| 安装到硬盘后无法启动 | 检查 BIOS 启动顺序、是否 UEFI/Legacy 与安装时一致；若有双系统，确认 Grub 是否识别到新系统。 |
| Live 下 eggs krill 报错 | 确认目标磁盘分区选择正确、无重要数据或已备份；可改用 Calamares 图形安装再试。 |

## 七、安全提醒

- 首次登录后建议修改 **root** 与 **amyclaw** 的默认密码。
- API Key、Token 等仅通过管理页或本机 `.env` 配置，不要提交到版本库或写入镜像。
- 若管理页对外网开放，务必配合防火墙、强密码，并考虑 HTTPS 与访问限制。

## 八、流程速查（无图形界面）

```text
1. 获取 ISO（如 amyclaw_v1.01_20260316_0601.iso）
2. 用 Ventoy/dd 写入 U 盘，或配置 PXE
3. 目标机从 U 盘/PXE 启动 → 进入 Live
4. Live 终端执行：sudo eggs krill → 按提示选磁盘完成安装
5. 重启并从硬盘启动
6. 执行：
   sudo mkdir -p /root/.openclaw
   sudo cp -r /opt/amyclaw/openclaw/production-template/state/* /root/.openclaw/
   sudo cp /root/.openclaw/.env.example /root/.openclaw/.env 2>/dev/null || true
   sudo systemctl enable --now openclaw-gateway openclaw-management
7. 浏览器打开 http://<本机IP>:8080 → 设置 Token 与各项配置
```

完成以上步骤后，全新机器即可作为一台带 OpenClaw Gateway 与管理前端的独立主机使用。
