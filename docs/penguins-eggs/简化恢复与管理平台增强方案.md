# 简化恢复与管理平台增强方案

目标：**恢复过程极简、标准化一键恢复，部署与日常使用仅通过管理平台网页完成**；系统密码可在安装时跳过，在管理平台统一修改并做安全提示。

---

## 一、目标体验（最终形态）

### 1.1 恢复阶段

| 步骤 | 当前 | 目标 |
|------|------|------|
| 获取 ISO、U 盘启动 | 同左 | 不变 |
| 安装到硬盘（eggs krill / Calamares） | 可能要求填 root/用户密码 | **仅需输入操作系统用户名与密码（root、普通用户）；密码可留空，选择「稍后在管理平台设置」** |
| 首次进入系统 | 需在终端执行多条命令：初始化 state、enable 服务等 | **全部默认一键完成**：首次开机自动完成 state 初始化、服务启用，无需用户敲命令 |
| 后续配置与使用 | 部分在终端、部分在管理页 | **全部在管理平台网页完成** |

### 1.2 日常使用

- **仅使用管理平台**：打开 `http://<本机IP>:8080`，完成 Gateway Token、飞书/企业微信/模型、以及**系统账号密码修改**等所有操作。
- 无需再登录 SSH 或终端做部署类操作（除非检修故障）。

---

## 二、安装器侧：用户名与密码可选

### 2.1 行为约定

- 安装时（eggs krill 或 Calamares）只多问两项：
  - **root 密码**：可输入，也可选择「不设置 / 稍后在管理平台设置」。
  - **普通用户（如 amyclaw）密码**：同上。
- 若用户选择「稍后设置」，安装器写入默认密码（与现有一致，如 `amyclaw20260315`），并在系统中做标记（例如某文件或标志），供管理平台识别「需提示用户修改密码」。

### 2.2 实现要点（供后续开发）

- **eggs krill**：若支持交互，增加「跳过密码 / 使用默认并在首次通过管理平台修改」的选项；若不支持，可保持当前默认密码，完全依赖管理平台提示修改。
- **Calamares**：在用户模块中允许密码为空或“首次登录必须修改”，并与“管理平台首次提示”语义一致。

---

## 三、首次开机：一键完成恢复配置

### 3.1 目标

- 用户从 ISO 安装到硬盘并重启后，**不再需要**在终端执行任何命令。
- 以下在首次启动时由 **systemd 或一次性脚本** 自动完成：
  - 创建并初始化 `OPENCLAW_STATE_DIR`（如 `/root/.openclaw`），从 `/opt/amyclaw/openclaw/production-template/state/` 拷贝模板。
  - 若不存在 `.env`，从 `.env.example` 复制一份。
  - 启用并启动 `openclaw-gateway`、`openclaw-management`。
- 用户只需：**打开浏览器访问管理平台**。

### 3.2 实现要点

- 在镜像中增加一个 **firstboot / 首次配置** 机制，任选其一：
  - **systemd oneshot 服务**：在 `multi-user.target` 之后运行一次，执行上述初始化并 `systemctl enable openclaw-gateway openclaw-management` 且 start，执行后禁用自己或删除标记，保证只跑一次。
  - **live 或安装后脚本**：Calamares/krill 的 hook 或 post-install 脚本在安装完成时写入「首次启动时执行」的脚本，由 systemd 或 rc.local 调用一次。
- 保证幂等：若 state 目录已存在且非空，可跳过拷贝或只补全缺失文件，避免覆盖用户已改配置。

## 四、管理平台增强：系统密码修改与安全提示

### 4.1 新增「系统安全」或「系统设置」区块

在管理平台（8080）中增加一个**仅登录后可见**的区块，例如「系统安全」或「系统设置」，包含：

1. **修改 root 密码**
   - 输入当前 root 密码（可选，视实现方式）+ 新密码 + 确认新密码。
   - 提交后，后端调用系统接口（如 `chpasswd` 或 `passwd`）完成修改。
   - 提示：仅限 root 或具有相应权限的进程可修改 root 密码；管理服务需以 root 运行或通过 sudo/helper 调用。

2. **修改普通用户（如 amyclaw）密码**
   - 同上，选择或固定用户名为 `amyclaw`，输入新密码 + 确认。
   - 后端同样通过 `chpasswd` 或 `passwd` 完成。

3. **安全提示与「不再提示」**
   - **首次或使用默认密码时**，在页面上方显示醒目提示条，例如：
     - 「为保障系统安全，请尽快修改 root 与 amyclaw 的默认密码。」
     - 提供按钮「前往修改」跳转到上述密码修改表单项。
     - 提供选项「已修改，不再提示」：用户勾选并确认后，**记住该状态**（见下），以后不再显示该提示条。
   - **记住方式**：在管理端可写目录（如 `data/`）下持久化一个标志文件或配置项，例如 `data/password-reminder-dismissed.json` 或 `security.json` 中 `passwordReminderDismissed: true`，仅用于前端是否展示提示，不存密码。

### 4.2 后端接口约定（建议）

- `POST /api/system/change-password`（或类似路径）  
  - Body：`{ "user": "root" | "amyclaw", "currentPassword": "...", "newPassword": "..." }`  
  - 校验：需已通过管理平台 Token 认证；可选校验 currentPassword（若实现校验则更安全）。  
  - 实现：使用 `child_process.spawn("chpasswd")` 或调用 `passwd` 的非交互模式（若可用），以 root 运行管理服务时可行。
- `GET /api/system/password-reminder-status`  
  - 返回：`{ "showReminder": true|false }`。  
  - 逻辑：若存在「已修改，不再提示」标志则 `showReminder: false`，否则为 `true`（或结合「是否仍为默认密码」逻辑，若可检测）。
- `POST /api/system/dismiss-password-reminder`  
  - 用户点击「已修改，不再提示」后调用，写入上述标志，使后续 `showReminder` 为 false。

### 4.3 前端行为

- 页面加载时请求 `GET /api/system/password-reminder-status`，若 `showReminder === true` 则显示顶部提示条。
- 提供「前往修改」与「已修改，不再提示」按钮；后者调用 `POST /api/system/dismiss-password-reminder` 后隐藏提示并记住。
- 密码修改表单在「系统安全」区块内，提交时调用 `POST /api/system/change-password`，成功后可提示「请牢记新密码」。

### 4.4 安全与权限

- 修改系统密码属于敏感操作，必须：
  - 仅允许已通过管理平台 Token（或现有登录态）认证的请求；
  - 管理服务以 root 运行，或通过受控的 setuid/sudo 脚本调用 `chpasswd`，并严格限制脚本仅能被管理服务调用。
- 不在前端或日志中记录、不写入配置文件明文密码；`chpasswd` 通过 stdin 传参后立即结束，避免长时间保留在内存中（按系统最佳实践）。

## 五、标准化与「一键恢复」小结

- **安装**：只问操作系统用户名与密码；密码可不输入，留到管理平台修改。
- **首次开机**：自动完成 state 初始化与服务启用，无需用户执行任何命令。
- **部署与使用**：仅通过管理平台网页完成，包括：
  - Gateway Token、飞书/企业微信/模型/向量等业务配置；
  - **系统密码修改**；
  - **首次安全提示**（修改默认密码）及「已修改，不再提示」的记住，确保系统安全。

这样，恢复流程收敛为：**装系统（可选填密码）→ 开机 → 打开管理平台 → 按需改密码并关闭提示 → 配置业务**，实现「只靠管理平台」的标准化一键恢复与日常使用。
